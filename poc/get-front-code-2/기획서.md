# 프론트엔드 프로젝트 자동 분석기 구축 기획서 (v5 - 플러그인 아키텍처 적용 최종본)

## 1. 개요
본 시스템은 **폐쇄망 환경(인터넷 미연결)**에서 동작하며, 프론트엔드 프로젝트 코드를 정적 분석(Static Analysis)하여 **"화면별 호출 API 매핑 목록"**을 자동으로 역추적하고 추출하는 도구입니다.

---

## 2. 분석 대상 및 타겟 선정 근거
시장의 프론트엔드 생태계 점유율과 개발 리소스의 '선택과 집중'을 고려하여, **React 및 Next.js 생태계를 1차 타겟(Phase 1)**으로 확정하였습니다.

| 프레임워크 | 시장 점유율 | 지원 여부 |
| :--- | :--- | :--- |
| **React** (Next.js 포함) | 약 80% | ✅ **Phase 1 (우선 개발)** |
| **Vue 3** | 약 14% | ⏳ Phase 2 (React 완료 후 추가) |
| Svelte / Angular 등 | 약 6% | 향후 확장 |

가장 파급력이 큰 React(80%)를 단기 POC 목표로 삼아 성공 여부를 빠르게 검증(Fail-fast)합니다.

### 2.1. 스코프 아웃 (기술적으로 명시 제외되는 항목)
| 항목 | 사유 | 대응 계획 |
| :--- | :--- | :--- |
| **GraphQL** (Apollo 등) | REST와 AST 패턴이 근본적으로 달라 별도 파서 로직 필요 | Phase 2 이후 재검토. Phase 1은 "REST API만 대상" |

## 3. 도입 예정 오픈소스 및 라이브러리 목록
본 분석기 개발 파이프라인에 사용될 핵심 라이브러리입니다. (모두 폐쇄망 로컬 실행 가능)

| 라이브러리 명 | 분류 | 역할 및 도입 목적 |
| :--- | :--- | :--- |
| **`@babel/parser`** | AST 파서 | JavaScript/TypeScript 코드를 메모리 상의 AST 객체로 파싱 |
| **`@babel/traverse`** | AST 횡단/추적 | 생성된 AST 트리를 순회하며 특정 함수 호출 구문이나 Import 연결 고리 추적 |
| **`@babel/types`** | AST 헬퍼 | AST 노드의 타입을 안전하고 정확하게 판별(`t.isCallExpression()`)하여 구현의 견고함 향상 |
| **`enhanced-resolve`** | 모듈/경로 해석기 | `tsconfig` 등에 설정된 복잡한 절대경로(Alias, 예: `@/api`)를 물리적 파일 경로로 치환 |
| **`fast-glob`** | 파일 스캐너 | 지정된 프레임워크 라우터 규칙(`src/pages/**` 등)에 부합하는 대상 파일을 초고속 스캔 |

## 4. 핵심 아키텍처: 플러그인 기반 다중 레이어 구조
본 분석기의 핵심은 프레임워크 의존성을 제거하는 **Adapter 레이어**와, 데이터 페칭 라이브러리 의존성을 제거하는 **Plugin(Resolver) 레이어**를 완벽하게 분리한 확장성 높은 아키텍처입니다.

### 아키텍처 구조도 (v5 기준)
```text
[1. 프레임워크 판별 및 Adapter 로드] ── package.json 분석
      │  ├─ Next Adapter (app/**/page.tsx 등)
      │  └─ React Adapter (src/pages/** 등)
      ▼
[2. 오케스트레이터 (Analyzer)] 
      │  package.json의 의존성(dependencies)을 분석하여 최적화된 Resolver 플러그인 동적 로드
      │  (예: @reduxjs/toolkit이 있으면 RTK Query 로드, 없으면 스킵 + Axios는 기본 로드)
      │  이후 로드된 플러그인들의 초기화(init) 수행 (URL 사전 생성 등)
      ▼
[3. AST 파싱 및 순회 (Traverser)] 
      │  Adapter가 찾은 UI 컴포넌트들을 하나씩 AST로 변환 후 순회
      ▼
[4. 플러그인 기반 데이터 페칭 추출 (Hook Resolvers)] ── 플러그인 체인 ──┐
      │  Traverser는 함수 호출을 발견하면 '현재 로드된' 플러그인에 매핑을 위임 │
      │  ├─ RTK Query Resolver (해당 라이브러리 사용 시에만 활성화)      │
      │  ├─ React Query Resolver (해당 라이브러리 사용 시에만 활성화)    │
      │  ├─ SWR Resolver (해당 라이브러리 사용 시에만 활성화)            │
      │  └─ Axios/Fetch Resolver (기본 활성화 - 순수 HTTP 클라이언트)    │
      ▼                                                                 ▼
[최종 산출물] 화면 ↔ API 매핑 마크다운 리포트 생성
```

## 5. 데이터 페칭 라이브러리 지원 (Plugin Resolvers)
엔진(Traverser)은 특정 라이브러리의 문법을 직접 알지 못합니다. 대신 4개의 핵심 플러그인(Resolver)이 각각의 패턴을 전담하여 추적합니다.
모든 플러그인 구현체는 프로젝트 내 **`src/core/resolvers/`** 디렉토리에 캡슐화되어 관리됩니다.

### 5.1. RTK Query Resolver (Redux Toolkit)
* **문제 상황 (특징)**: React Query나 Axios는 화면 파일(예: `Dashboard.tsx`)에 URL(`/api/users`)이 직접 적혀 있지만, RTK Query는 화면 파일에 자동으로 생성된 훅 이름(`useGetUsersQuery()`)만 덩그러니 존재합니다. 진짜 URL은 별도의 설정 파일(`.api.ts`)에 숨겨져 있어 화면 파일만 스캔해서는 URL을 알아낼 수 없습니다.
* **해결 및 동작 방식 (사전 생성 전략)**: 
  1. 엔진이 분석을 시작하기 전 `init()` 단계에서 프로젝트 내 모든 API 파일(`.api.ts`)을 선행 스캔합니다.
  2. "아하, `getUsers` 엔드포인트는 `/api/users`로 가고, 화면에서 쓸 때는 `useGetUsersQuery`라는 훅을 쓰겠네?" 하고 메모리에 **단어장(사전, Dictionary)**을 만들어 둡니다. (`{ "useGetUsersQuery": "/api/users" }`)
  3. 이후 화면 분석 시 `useGetUsersQuery`를 발견하면, 만들어둔 단어장을 조회하여 URL을 1:1로 매핑해냅니다.

### 5.2. React Query Resolver (TanStack Query)
* **특징**: 컴포넌트 내부에 `useQuery`, `useMutation` 등을 선언. (v3 배열 형태, v5 객체 형태 혼재)
* **동작 방식**: `useQuery({ queryKey: ['/users'] })` 패턴에서 첫 번째 인자의 객체 속성을 분석하거나, `useQuery(['/users'])`의 배열 첫 번째 요소를 분석하여 엔드포인트와 GET/POST 여부를 유추합니다.

### 5.3. SWR Resolver (Vercel)
* **특징**: `useSWR('/api/users')` 같이 키 자체가 URL 역할을 수행.
* **동작 방식**: 첫 번째 인자가 문자열이거나 배열인 경우 이를 엔드포인트로 인식하고, 기본적으로 `GET` 메서드로 취급합니다.

### 5.4. Axios & Fetch Resolver
* **특징**: 원시적인 HTTP 클라이언트 호출.
* **동작 방식**: `axios.get()`, `axios.post()` 등의 프로퍼티 이름에서 메서드를 추출하거나, `fetch(url, { method: 'POST' })`의 옵션 객체를 분석하여 메서드와 URL을 추출합니다.

## 6. 동적 URL 및 노이즈 필터링 전략
* **동적 URL 정규화**: `axios.get(\`/users/\${userId}\`)` 코드는 변수 구간을 템플릿 리터럴로 파싱하여 `/users/{userId}` 와 같은 정규화된 표현식으로 치환합니다.
* **False Positive 제거**: `Sentry.init()`, `gtag()` 등 API 통신과 무관한 외부 도메인 및 SDK 함수 호출은 블랙리스트 필터를 거쳐 매핑 결과에서 배제됩니다.

---

## 7. 파싱 실패 처리 전략 (Fail-safe)
특정 파일 내에 알 수 없는 문법 오류나 분석 불가능한 패턴이 있어 파싱이 실패하더라도 전체 분석 파이프라인이 크래시(Crash)되지 않습니다. `try-catch`로 **Skip** 처리한 뒤 에러 로그를 남기고 다음 파일 분석을 이어갑니다.

## 9. 기대 효과
* **궁극의 확장성**: '플러그인(Plugin)' 아키텍처를 도입하여 향후 Jotai, Apollo, RTK Query v2 등 어떠한 새로운 데이터 라이브러리가 등장해도 **코어 엔진 수정 없이 Resolver 클래스 1개만 추가**하면 대응이 가능합니다.
* **정확도 향상**: 복잡한 RTK Query의 '선언과 사용' 분리 패턴을 완벽히 추적해 냄으로써, 기존 도구들이 분석하지 못하던 엔터프라이즈급 프로젝트(`davis-frontend` 등)의 API 호출 흐름도 100% 매핑해 냅니다.
* **프레임워크 독립성**: Next.js 든 순수 React 든 Adapter 계층이 라우팅을 알아서 해석해 주므로 단일 시스템으로 여러 생태계를 포용합니다.
